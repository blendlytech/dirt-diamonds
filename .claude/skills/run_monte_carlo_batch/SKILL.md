# run_monte_carlo_batch

**Purpose:** Tests the baseball simulation engine's statistical accuracy in isolation. Mandatory after **every** change to `AtBatResolver.cs` or the PED modifier weights (CLAUDE.md §Required Workflow Procedures).

**Execution:**

```powershell
dotnet run --project Tools/MonteCarloHarness -c Release
```

Optional: `-- --pa <count>` raises the Monte Carlo batch size (default 100,000; floor 10,000), `-- --repo <path>` if run outside the repo tree.

**What it proves (exit code 0 = all pass):**

1. **§5 numeric fixtures** — the resolver reproduces the worked examples in `docs/design/baseball_pa_outcome_model.md` (average baseline, elite slugger, ace) to ±1e-3, plus the §6 PED clamp (`min(100, round(power×1.5))`), seed determinism, and zero heap allocation per PA.
2. **§8 batch acceptance** — ≥10k average-vs-average PAs land inside every acceptance range (AVG .240–.260, OBP .308–.325, SLG .395–.430, OPS .710–.745, K% 20–25, BB% 7.5–10.5, HR/PA 2.7–3.7) and prints the slash line.
3. **Full pipeline** — two complete headless seasons on a scratch db (LeagueGenerator → TimeManager/EventBus → LeagueSimulator → season flush → StatsNormalizer), league-wide slash + R/G 4.2–4.8 per season, batting/pitching ledger consistency, stored rates match recomputation, integrity/FK checks.
4. **Flat GC profile** — a warmed game day allocates zero bytes.
5. **Micro-sim analytics (micro doc §3–§5)** — all seven 25×25 A_e matrices row-stochastic and equivalent to the runtime advancement functions, RE24 anchor from the fundamental matrix (empty/0-out in 0.42–0.56, ordering sane), the §5.2 count-chain absorption pinned to `AtBatResolver.ComputeProbabilities` analytically (≤1e-9), sampled per-PA neutral consistency, pitches/PA 3.7–4.0, zero-alloc pitch chain.
6. **Micro-sim attended games (micro doc §11)** — 2000 neutral-policy games (fatigue off) land inside the macro §8 ranges (test 2); late-inning OPS rise +.030–.150 and R/G uplift with fatigue on (test 4); PED capacity fixture 70→105→120 + post-game costs (test 5); bit-for-bit game determinism and zero-alloc warm attended games incl. human PAs (test 6); Game_Logs play-by-play + additive box-score flush.
7. **Career wiring (Phase 5)** — avatar creation (roster stays 9+5, displaced player benched to free agency, Game_State records the id, reboot restores it); a full autopilot season through the real event loop where the macro sim skips the attended team's pairing and the micro-sim plays it: every game played exactly once (GS = 2×games, W = L = games), macro cycle flushes + micro per-game flushes compose on the same season rows (ledgers agree, league line stays in §8), avatar season PA 450–800; the `PlayerIntentBridge` handshake drives a full interactive game from a scripted UI thread; a cancelled interactive game unwinds unflushed and is forfeited to the autopilot on the next tick.
8. **Heir mechanics — genetic blending & interest (`docs/design/heir_mechanics.md` §7/§8 checks 1–6, `HeirGenetics.cs` + `CareerManager.ConceiveChild`)** — §7 fixtures F1/F2/F4 reproduce exactly under an injected `bell` (F1 also checked as a full seven-rating vector); regression identity at `bell=0` (`child_r == round(50 + Heritability·(midparent−50))`) holds for both two-parent and lone-parent sweeps regardless of `Spread`; every blended rating and rolled interest stays in [0,100] across extreme parents/affinity × extreme bell; the lottery is mean-zero over 100k draws; the `interest ≥ 40` willing/unwilling gate classifies correctly at the exact rounding boundary; the §1.2 direction invariant (the older `Child`-edge endpoint is always the parent) holds across a birth-age/parent-age sweep through a real `ConceiveChild` call. **F3 note:** the design doc's "both parents 55 → 72" fixture is unreachable under the shipped model (`Spread=12` bounds the lottery to a max swing of ±12 off `blended=52.5`, so the ceiling is <64.5) — adapted into a qualitative "lucky roll beats blended" check rather than silently faked; flagged back to the doc's author.
9. **Lineage & succession (`docs/design/heir_mechanics.md` §5/§6/§8 checks 7–11, `CareerManager.Succeed`/`EvaluateSuccession` + the yearly aging tick)** — the 3-generation exit-criteria run through the real event loop: founder → heir → heir across two `SeasonRolledOverEvent` rollovers, `dynasty_generation` reaching 3, the aging tick incrementing every player exactly +1 per rollover; the same-role slot-inherit handoff and the role-mismatch displace-and-backfill handoff both preserve 9+5+3 on every team; the retiree is benched to FA with stats intact and (once past 42) is never signable again; GS/W/L identities hold across both succession seasons; **the generation-3 season lands inside the §8 AVG/OBP bands (the "no band moved" check)**; the lone-parent bloodline regresses toward the mean (90 → ≤73 → within ±20 of 50, no upward ratchet); a cold reload rehydrates generation/founder/avatar/`Child` edges mid-lineage. Plus the failure ladder (`NoHeirs`/`NoWillingHeir`/`NoPlayableHeir` each persisting `lineage_over_reason`, game-over mutating nothing else, aging continuing but succession parked after game-over), the health_ceiling≤40 retirement trigger driven through a real `ApplyPedGameCosts` erosion, and the generated-filler backfill when the free-agent pool is empty. **Fixture rule:** age-seeded fixtures must age the *whole* bloodline consistently — the §1.2 direction invariant (parent strictly older) is maintained by the shared aging tick in real play, and a fixture that ages only one member can make a retired ancestor resolve as an eligible "child".

10. **HS-6 person→performance modifiers (`high_school_person_layer.md` §6, `PersonEffects.cs`)** — §6.1 `PointsFor` pinned (zero-at-50 for every cap, ±cap endpoints, away-from-zero rounding fixtures, 0–100 monotone odd-symmetric sweep at caps 2/3); the §6.3 bake order pinned at both clamps (tier delta first, then person delta) plus the 0-person-delta identity to the exact pre-HS-6 tier shift; `BakeTeamDefense` neutral identity + rounded-lineup-mean teamwork fixtures; analytic per-lever direction against the unchanged resolver (power +2 → HR↑, contact +2 → hits↑, stuff +2 → K↑, control +2 → BB↓, defense +2 → in-play hits↓); same-seed full macro seasons proving **no source ≡ wired-but-rowless ≡ all-neutral rows** (bit-identical totals — why the MLB bit-identity guard survives HS-6 by contract, not luck) while a maxed team diverges yet keeps the league line (ΔAVG ≤ .003, ΔR/G ≤ .05 vs same-seed control); the live-world-shaped §2.5 organic-person-spread world diverging its same-seed control inside the same delta bound (the sanctioned re-band statement); the micro path replaying the identity ladder and a maxed-team divergence per game; and the teamwork-only defense byte pinned +2 through `TeamDefenseFor` in BOTH sims (a lone ±2 defense shift is too small to reliably diverge one game, so the byte is pinned exactly instead). Section (i) covers the **PersonLedger per-game refresh** (the HS-6 follow-up, `PersonLedger.cs` + `MicroGame.RefreshPersonCache`): a wired-but-empty ledger is bit-identical (empty = the Initialize bake stands, the no-boot-seed identity), announcements equal to the baked rows are per-slot no-ops (a survivor from before a re-Initialize beat is redundant, never stale), an in-season lever collapse announced on the bus lands the exact same-seed game a fresh re-Initialize produces (refresh ≡ re-Initialize), and the defense byte moves through the bus alone (neutral ±2 both directions, reinit parity). Refresh game divergence is deliberately not asserted (the teamwork test-power lesson — the byte pins plus the equality statement are the deterministic form).

**Tuning workflow (macro doc §8 / micro doc §11, change one block then re-run):** slash line off → §2 baselines; stars/scrubs spread wrong → §4.2 sensitivities; a skill inert/overpowered → §4.1 weights; runs off but slash fine → `BaseOutAdvancement.SingleScoresFrom2nd` / `DoubleScoresFrom1st`; micro pitches/PA off → `PitchChain.FoulShareOfStrikes`; fatigue too weak/strong → `PitcherFatigue` curve then capacity knobs; human input feel → `PitchChain` gains / `PlayerInputModel` (gated on the neutral-consistency tests staying green); dynasties drifting toward 100 or fading too fast → `HeirGenetics.HeirGeneticsProfile.Heritability`/`Spread` (re-check the §3 drift table by hand — this is the calibration-safety proof, not just a feel knob); heirs too eager/reluctant to play → `InterestBaseline`/`InterestAffinityMax`/`InterestSpread`/`InterestPlayThreshold`.
